# The Deep Thought Blueprint: Native TLS Decryption Engine in C#

## Overview
This document serves as the architectural specification and technical blueprint for a future AI entity embarking on the monumental task of constructing a fully native, 100% C# TLS 1.2 and TLS 1.3 decryption engine. 

The goal of this engine is to ingest two inputs:
1. A raw `.pcap` or `.pcapng` packet capture file.
2. A populated `SSLKEYLOGFILE` generated by a modern browser.

The required output is the fully reassembled, plaintext Application Layer data stream (e.g., HTTP/2, raw JSON) stripped of all cryptographic encapsulation, without any reliance on external binaries like Wireshark or `tshark.exe`.

## đźŹ—ď¸Ź Phase 1: TCP Stream Reassembly Engine
Before any cryptography can occur, the chaotic nature of raw network packets must be tamed. The engine must first reconstruct the logical, bidirectional data streams exactly as the application layer experienced them.

### Requirements:
1. **Packet Parsing (`SharpPcap`)**: Utilize `PacketDotNet` to strip Ethernet, IPv4/IPv6, and TCP headers, isolating the raw TCP payload arrays.
2. **Connection Tracking (The 5-Tuple)**: Map every packet into a distinct "Flow" defined by `Protocol`, `Source IP`, `Source Port`, `Destination IP`, and `Destination Port`. 
3. **Sequence & Acknowledgment Tracking**: 
   - Network packets do not arrive sequentially. The engine must implement a robust buffering system to hold out-of-order `TCP Sequence` segments.
   - It must handle **TCP Retransmissions** (dropping duplicates) and **Packet Loss** (halting decryption until the missing sequence arrives, or abandoning the flow).
4. **Output**: Give the cryptography layer a continuous, unfragmented byte stream for each direction (Client-to-Server, Server-to-Client) per flow.

## đź”’ Phase 2: TLS Protocol State Machine
The core complexity begins here. TLS records sit on top of the reassembled TCP stream. The engine must parse the TLS Record Layer formats and maintain a rigorous State Machine for every tracked flow.

### Record Layer Parsing:
Every TLS record begins with a 5-byte header:
- `Byte 0`: Content Type (e.g., `22` = Handshake, `23` = Application Data, `20` = ChangeCipherSpec).
- `Bytes 1-2`: Legacy Version (usually `0x0303` for TLS 1.2+).
- `Bytes 3-4`: Length of the fragment.

### The Handshake State Machine:
To decrypt `Application Data` (`Type 23`), the engine must actively monitor `Handshake` (`Type 22`) records to extract critical cryptographic inputs.

**Critical Extraction Targets:**
1. **Client Random (32 bytes)**: Sourced from the `ClientHello` message. *Crucial for matching the `SSLKEYLOGFILE`.*
2. **Server Random (32 bytes)**: Sourced from the `ServerHello` message.
3. **Session ID**: For tracking resumption.
4. **Cipher Suite selection**: Determines the exact AES/ChaCha math to use later.
5. **Key Share Extensions**: (TLS 1.3 only) For Elliptic Curve Diffie-Hellman (ECDHE) parameters.

## đź”‘ Phase 3: The `SSLKEYLOGFILE` Parser & Key Derivation (KDF)
The engine must parse the text-based key log file and marry it mathematically to the live network streams.

### Log File Ingestion:
The log file contains space-separated strings.
Example: `CLIENT_RANDOM 5C3A2B... 8A9B0C...`
- The engine must read this file into a high-performance hash map keyed by the `CLIENT_RANDOM` (the identical 32 bytes extracted from the `ClientHello` in Phase 2).

### Expanding the Master Secret (TLS 1.2):
Once a `CLIENT_RANDOM` match is found, the engine receives the `Master Secret` (48 bytes).
Using the `PRF` (Pseudo-Random Function) usually leveraging `HMAC-SHA256` or `HMAC-SHA384` (determined by the cipher suite), the engine must expand this:
```text
key_block = PRF(Master_Secret, "key expansion", Server_Random + Client_Random)
```
This output `key_block` is then carved up into the specific keys needed for the AES engines:
1. Client Write MAC Key
2. Server Write MAC Key
3. Client Write Encryption Key
4. Server Write Encryption Key
5. Client Write IV (Initialization Vector)
6. Server Write IV

### TLS 1.3 Key Schedule (The Nightmare Mode):
TLS 1.3 fundamentally changed decryption. There is no single "Master Secret." The `SSLKEYLOGFILE` provides entirely separate secrets for different phases:
- `CLIENT_HANDSHAKE_TRAFFIC_SECRET`
- `SERVER_HANDSHAKE_TRAFFIC_SECRET`
- `CLIENT_TRAFFIC_SECRET_0`
- `SERVER_TRAFFIC_SECRET_0`

The engine must implement the **HKDF (HMAC-based Extract-and-Expand Key Derivation Function)** (RFC 5869). It must literally derive independent Write Keys and Initialization Vectors on-the-fly for every state transition in the protocol.

## đź›ˇď¸Ź Phase 4: Authenticated Encryption with Associated Data (AEAD)
With the Write Keys and IVs mathematically derived uniformly against both sides, the engine can finally attack the encrypted `Type 23` payloads.

### Cipher Engine Implementation (`System.Security.Cryptography`):
The chosen `Cipher Suite` from Phase 2 dictates the math. Most modern web traffic uses either:
- **AES-128-GCM** / **AES-256-GCM** (Galois/Counter Mode)
- **ChaCha20-Poly1305**

The AI must build wrappers around C#'s `AesGcm` and `ChaCha20Poly1305` classes.

### The Cryptographic Nonce (The Silent Killer):
AEAD ciphers require a completely unique, never-repeated `Nonce` (Number Used Once) for every single TLS record decrypted. 
- In TLS 1.2, this is often explicitly transmitted on the wire alongside the packet.
- In TLS 1.3, it is **implicitly generated** by XORing the derived static IV with the 64-bit **TLS Sequence Number**.

**Critical AI Design Note:** 
The engine *must* maintain a strict, 0-indexed integer counter (`Sequence Number`) for every TLS record sent and received in the flow. If the Packet Reassembly engine in Phase 1 drops a packet, the sequence numbers will desynchronize, the Nonce will be mathematically wrong, and the entire `AesGcm.Decrypt()` call will instantly throw a fatal CryptographicException for the remainder of the session.
